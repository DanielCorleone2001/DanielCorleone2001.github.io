发表于 2020-12-30

并发BUG的三个源头

可见性

·一个线程对共享变量的修改，另外一个线程能够立刻看到，我们称为可见性

·单核时代，由于所有的线程都是在同一个CPU上执行的，CPU的缓存(cache)和内存数据的一致性其实很好解决。

因为一个进程对缓存的读写操作，对于其他的线程是具有可见性的。很简单的道理，二者都作用在CPU的缓存上，当某一线程改变了内存的值后让出使用权，其他线程再访问的时候是能正确的获取到相应的值。

·多核时代就不一样了，多核导致每个CPU都有自己的cache，对于维持缓存和内存数据的一致性就变得困难了，相应的也很容易产生bug : ( 这时候不同CPU上的线程是不具备可见性的

这段代码理论上执行出来输出的值应该是20k，但是实际上输出的值却没有达到20k。如果改成每次加1亿，执行出来的结果不出意外的话，1亿多一点点，导致这个问题的出现就是因为缓存的可见性问题

原因：假设线程 A 和线程 B 同时开始执行，那么第一次都会将 count=0 读到各自的 CPU 缓存里，执行完 count+=1 之后，各自 CPU 缓存里的值都是 1，同时写入内存后，我们会发现内存中是 1，而不是我们期望的 2。之后由于各自的 CPU 缓存里都有了 count 的值，两个线程都是基于 CPU 缓存里的 count 值来计算，所以导致最终 count 的值都是小于 20000 的。这就是缓存的可见性问题。循环 10000 次 count+=1 操作如果改为循环 1 亿次，你会发现效果更明显，最终 count 的值接近 1 亿，而不是 2 亿。如果循环 10000 次，count 的值接近 20000，原因是两个线程不是同时启动的，有一个时差。

执行这条语句，需要三步操作：

​ a.将变量的值从内存里面取出后，放到寄存器(count=0)

​ b.在寄存器中执行+1的操作(count+1=1)

​ c.将结果写入内存(当然cache的存在会导致写入的不是内存而是缓存)(count=1写入内存/cache)

三条语句才是不可再分，真正在计算机里实现的。但是bug为什么产生呢？这就涉及到了操作系统进行线程切换的知识。假设有m、n线程，count=0，当m执行完a指令后操作系统进行了线程切换到n线程，这时候n顺序执行完了a、b、c三条指令，在切换到m线程的b、c指令，最后的结果导致n线程虽然执行了，可是从结果来看，只有m线程的操作是有效的。当然有人会问，n执行完count不是等于1了吗？但是此时你得明白，m并没有从内存里再读count的值，所以m线程会认为count的值仍然是0，这就导致了两个线程都执行了+1操作，count并没有等于2，而是1的现象。这也就是CPU指令原子性和操作系统线程切换带来的恶心的bug。(现代的操作系统由于线程切换不需要进行内存地址映射，所以大多都是轻量级的线程切换而不是进程切换，所以本文只讨论这一问题)

有序性

编译器有时候为了会优化性能，理论上会顺序执行你的高级语言程序，其实并不是。有时会调整高级语言执行的顺序却不影响程序的执行结果，但是这同时也导致了某些和设想之中不一样的bug。

例如有m、n进程调用方法getInstance来初始化实例。理论上是这么个情况：

二者都发现instance==null然后对Singleton.class上锁。但是JVM会只让一个线程成功上锁，假设是m成功上锁，这时候m就会继续发现instance==null，然后初始化instance，然后释放锁。

这时候n线程发现instance！=null了，不能上锁了，从而实现了只创建一个单例。

但是我们认为线程n在进行new操作的过程是这样的：

​ a.分配一块内存T

​ b.在T上初始化单例对象

​ c.T的地址赋值给instance

实际上却是这样的：

​ a.分配一块内存T

​ b.T的地址赋值给instance

​ c.在T上初始化单例对象

这时候问题就出来了，假设m执行完了实际上的指令b，线程切换到n，然后n检查instance是否为空，这时候instance是有T的地址的，肯定是非空，所以线程n就会直接返回instance实例。但是呢，m却还没对其初始化，这也就会出现空指针异常。

小拓展

在 32 位的机器上对 long 型变量进行加减操作存在并发隐患？

答：这是对的。因为由于原子性的存在，在进行线程切换的时候会产生某些不可名状的bug。32位对于long类型分成了两份来处理，有可能某线程在处理高位字节(前4位)，这时候发生了线程切换，另一个线程把低位的给改了。。。这时候就会导致加减法操作出现错误。

解决方法：把long/double 变量声明为volatile或是同步加锁synchronize以避免并发问题
