并发编程

并发编程的核心问题

分工

·概念：如何高效地拆解任务并分配给线程

·举例：Fork/Join框架

同步

·概念：线程之间怎么协作运行

·举例：CountDownLatch

互斥

·概念：保证同一时刻只有一个线程访问共享资源(操作系统里的知识)

·举例：可重入锁

个人思考

这三者其实是层层递进的关系，分工导致了任务的拆解，进而需要线程间的通信，从而引入了同步。而引入同步之后，线程对于临界资源的访问具有互斥性，从而引进了互斥。

并发BUG的三个源头

可见性

·一个线程对共享变量的修改，另外一个线程能够立刻看到，我们称为可见性

·单核时代，由于所有的线程都是在同一个CPU上执行的，CPU的缓存(cache)和内存数据的一致性其实很好解决。

因为一个进程对缓存的读写操作，对于其他的线程是具有可见性的。很简单的道理，二者都作用在CPU的缓存上，当某一线程改变了内存的值后让出使用权，其他线程再访问的时候是能正确的获取到相应的值。

·多核时代就不一样了，多核导致每个CPU都有自己的cache，对于维持缓存和内存数据的一致性就变得困难了，相应的也很容易产生bug : ( 这时候不同CPU上的线程是不具备可见性的

这段代码理论上执行出来输出的值应该是20k，但是实际上输出的值却没有达到20k。如果改成每次加1亿，执行出来的结果不出意外的话，1亿多一点点，导致这个问题的出现就是因为缓存的可见性问题

原因：假设线程 A 和线程 B 同时开始执行，那么第一次都会将 count=0 读到各自的 CPU 缓存里，执行完 count+=1 之后，各自 CPU 缓存里的值都是 1，同时写入内存后，我们会发现内存中是 1，而不是我们期望的 2。之后由于各自的 CPU 缓存里都有了 count 的值，两个线程都是基于 CPU 缓存里的 count 值来计算，所以导致最终 count 的值都是小于 20000 的。这就是缓存的可见性问题。循环 10000 次 count+=1 操作如果改为循环 1 亿次，你会发现效果更明显，最终 count 的值接近 1 亿，而不是 2 亿。如果循环 10000 次，count 的值接近 20000，原因是两个线程不是同时启动的，有一个时差。

有序性

编译器有时候为了会优化性能，理论上会顺序执行你的高级语言程序，其实并不是。有时会调整高级语言执行的顺序却不影响程序的执行结果，但是这同时也导致了某些和设想之中不一样的bug。

例如有m、n进程调用方法getInstance来初始化实例。理论上是这么个情况：

二者都发现instance==null然后对Singleton.class上锁。但是JVM会只让一个线程成功上锁，假设是m成功上锁，这时候m就会继续发现instance==null，然后初始化instance，然后释放锁。

这时候n线程发现instance！=null了，不能上锁了，从而实现了只创建一个单例。

但是我们认为线程n在进行new操作的过程是这样的：

​ a.分配一块内存T

​ b.在T上初始化单例对象

​ c.T的地址赋值给instance

实际上却是这样的：

​ a.分配一块内存T

​ b.T的地址赋值给instance

​ c.在T上初始化单例对象

这时候问题就出来了，假设m执行完了实际上的指令b，线程切换到n，然后n检查instance是否为空，这时候instance是有T的地址的，肯定是非空，所以线程n就会直接返回instance实例。但是呢，m却还没对其初始化，这也就会出现空指针异常。

小拓展

在 32 位的机器上对 long 型变量进行加减操作存在并发隐患？

答：这是对的。因为由于原子性的存在，在进行线程切换的时候会产生某些不可名状的bug。32位对于long类型分成了两份来处理，有可能某线程在处理高位字节(前4位)，这时候发生了线程切换，另一个线程把低位的给改了。。。这时候就会导致加减法操作出现错误。

解决方法：把long/double 变量声明为volatile或是同步加锁synchronize以避免并发问题

volatile

作用

保证内存的可见性，并禁止指令重排。

原子性

非原子操作就算使用volatile也不能保证原子性。

例如线程A取出i = 0想要进行自增，但是取出来后就被阻塞了。这时线程B取出i 进行自增，会发现i = 0.当线程A不在阻塞后才加一，B也加一。最后结果确是i经过两次线程的加一操作，却只等于1.解决方法，使用atomic类下的自增操作。(用CAS来实现的，底层的指令应该是CMPXCHG)

有序性

防止指令重排。

底层原理

使用volatile关键字，底层会在前面加上lock执行，提供了内存屏障。

内存屏障：

1.确保指令不会重排。执行这条指令时，前面的指令已经全部执行

2.强制对前面的操作写入内存中

Synchronized

锁

以前是重量级锁。

jdk1.6后引入了偏向锁和轻量级锁。

锁的状态：无锁->偏向锁->轻量级锁->重量级锁

锁能升级，不能降级。

无锁

无锁没有对资源进行锁定，所有的线程都能访问并修改同一个资源，但同时只有一个线程能修改成功。

无锁的特点就是修改操作在循环内进行，线程会不断的尝试修改共享资源。如果没有冲突就修改成功并退出，否则就会继续循环尝试。如果有多个线程修改同一个值，必定会有一个线程能修改成功，而其他修改失败的线程会不断重试直到修改成功。上面我们介绍的CAS原理及应用即是无锁的实现。无锁无法全面代替有锁，但无锁在某些场合下的性能是非常高的。

偏向锁

偏向锁是指一段同步代码一直被一个线程所访问，那么该线程会自动获取锁，降低获取锁的代价。

在大多数情况下，锁总是由同一线程多次获得，不存在多线程竞争，所以出现了偏向锁。其目标就是在只有一个线程执行同步代码块时能够提高性能。

当一个线程访问同步代码块并获取锁时，会在Mark Word里存储锁偏向的线程ID。在线程进入和退出同步块时不再通过CAS操作来加锁和解锁，而是检测Mark Word里是否存储着指向当前线程的偏向锁。引入偏向锁是为了在无多线程竞争的情况下尽量减少不必要的轻量级锁执行路径，因为轻量级锁的获取及释放依赖多次CAS原子指令，而偏向锁只需要在置换ThreadID的时候依赖一次CAS原子指令即可。

偏向锁只有遇到其他线程尝试竞争偏向锁时，持有偏向锁的线程才会释放锁，线程不会主动释放偏向锁。偏向锁的撤销，需要等待全局安全点（在这个时间点上没有字节码正在执行），它会首先暂停拥有偏向锁的线程，判断锁对象是否处于被锁定状态。撤销偏向锁后恢复到无锁（标志位为“01”）或轻量级锁（标志位为“00”）的状态。

偏向锁在JDK 6及以后的JVM里是默认启用的。可以通过JVM参数关闭偏向锁：-XX:-UseBiasedLocking=false，关闭之后程序默认会进入轻量级锁状态。

轻量级锁

是指当锁是偏向锁的时候，被另外的线程所访问，偏向锁就会升级为轻量级锁，其他线程会通过自旋的形式尝试获取锁，不会阻塞，从而提高性能。

在代码进入同步块的时候，如果同步对象锁状态为无锁状态（锁标志位为“01”状态，是否为偏向锁为“0”），虚拟机首先将在当前线程的栈帧中建立一个名为锁记录（Lock Record）的空间，用于存储锁对象目前的Mark Word的拷贝，然后拷贝对象头中的Mark Word复制到锁记录中。

拷贝成功后，虚拟机将使用CAS操作尝试将对象的Mark Word更新为指向Lock Record的指针，并将Lock Record里的owner指针指向对象的Mark Word。

如果这个更新动作成功了，那么这个线程就拥有了该对象的锁，并且对象Mark Word的锁标志位设置为“00”，表示此对象处于轻量级锁定状态。

如果轻量级锁的更新操作失败了，虚拟机首先会检查对象的Mark Word是否指向当前线程的栈帧，如果是就说明当前线程已经拥有了这个对象的锁，那就可以直接进入同步块继续执行，否则说明多个线程竞争锁。

若当前只有一个等待线程，则该线程通过自旋进行等待。但是当自旋超过一定的次数，或者一个线程在持有锁，一个在自旋，又有第三个来访时，轻量级锁升级为重量级锁。

重量级锁

升级为重量级锁时，锁标志的状态值变为“10”，此时Mark Word中存储的是指向重量级锁的指针，此时等待锁的线程都会进入阻塞状态。

底层原理

修饰变量：使用monitor enter 和 monitor exit指令

修饰方法：使用ACC_Synchronized指令声明即将进入方法体。

本质都是调用Monitor.
