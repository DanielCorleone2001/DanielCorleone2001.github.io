链表实现LRU缓存淘汰算法

发表于 2020-12-30

链表

性质

说到链表，很相似的就是数组。二者之间的区别其实很明显：

​ -数组能实现随机访问，而链表只能从头部开始慢慢往下找

​ -数组对于内存的要求很高：一定要是一段连续的空间；

​ -链表的话不要求是连续的内存空间，只要求满足大小即可

这么说下来，比如申请存储100个int类型的元素，数组会要求400个字节的连续内存空间，而链表需要800个字节的空间，连续不连续是无所谓的。为什么多了一倍的内存空间呢？因为链表还需存储地址。

分类

链表常见的结构有：单链表、双链表、循环链表

单链表：

​ –>data&next->–>data&next–>data&next->….->data&next->NULL

首和尾是比较特殊的节点：头节点用来记录链表的基地址，尾节点的next指向空来保证链表结构的完备性。

循环链表：

基本结构和单链表类似，只是尾节点指向头节点。用来解决约瑟夫问题很方便。

双向链表：

基本结构和单链表类似，每个节点都会多了个指向前驱节点的指针

那么单链表有两个域：数据域+指向后继节点的指针域

双向链表有三个域：数据与+指向后继节点的指针域+指向前趋节点的指针域

双向链表相较于单链表的一大优势在于：找到前驱节点的时间复杂度为O(1)

而单链表只能从头节点慢慢往下找，所以仍然是O(n).而且，对于插入和删除也是有优化的。可能会有问题：单链表的插入删除不是O(1)吗？是的，但是一般情况下，我们想要进行插入删除操作，很多时候还是得先进行查找，再插入或者删除，可见其实是先O(n),再O(1).

基本操作

插入：

​ 时间复杂度：O(1)

删除：

​ 时间复杂度O(1)

接着上面的问题。要解决疑问，就得先清除删除有哪些情况：

​ -删除节点中的值等于某个定值的节点

​ -删除给定指针指向的节点

第一种情况单向双向都只能老老实实先O(n),再O(1)，没有所谓的优化

第二种情况，单链表需要先获取目标节点的前驱节点，这时候找到的代价就是O(n),相应的双向链表只需要O(1)就行啦。

同理对应插入操作，不再解释。

查找：即访问索引为k的元素

链表实现LRU缓存淘汰算法

插播信息

CPU在从内存读取数据的时候，会先把读取到的数据加载到CPU的缓存中。而CPU每次从内存读取数据并不是只读取那个特定要访问的地址，而是读取一个数据块(这个大小我不太确定。。)并保存到CPU缓存中，然后下次访问内存数据的时候就会先从CPU缓存开始查找，如果找到就不需要再从内存中取。这样就实现了比内存访问速度更快的机制，也就是CPU缓存存在的意义:为了弥补内存访问速度过慢与CPU执行速度快之间的差异而引入。 对于数组来说，存储空间是连续的，所以在加载某个下标的时候可以把以后的几个下标元素也加载到CPU缓存这样执行速度会快于存储空间不连续的链表存储。

LRU

·LRU：least recent used ，该算法找到最近最少使用的东西。

·应用：

操作系统虚拟存储器中就是用到了这个算法。虚拟存储器中，每个进程只会被分配几个内存块。当内存块可使用的块已经满了，就得从磁盘重新读取一个内存块的数据量。此时得使用替换算法来决定要去掉哪一个内存块。

算法思想

使用有序单链表，注意这里的有序不是值按照大小顺序，而是按我们定义的顺序：越靠进头节点的就是约常被使用的，越靠近尾节点的就是越不常使用的。这时候就很明朗了：

当有一个新的数据要被访问的时候，从头结点开始访问链表，

​ -如果找到了该元素对应的节点，那么说明这个元素即将要被使用，就要删除这个元素所在的位置，插入到头节点

​ -如果找不到的话，那就说明不在缓存链表里面。这时候又得判断：

​ 如果缓存还有容量，则直接将该元素插入到链表的头部；

​ 如果没有了容量，那么就删除尾节点，再将元素插入到链表的头部

接着分析一下算法的复杂度：其实不管有没有找到数据，也不管缓存是否已经满了，都是要从头结点遍历一次，所以时间复杂度为O(n)。但这时候相信你们也发现了，其实hashtable(散列表)是它的优化版本。每次都记下数据的位置，那么不就解决了它每次都要遍历付出的高代价啦。

并发BUG的三个源头

可见性

·一个线程对共享变量的修改，另外一个线程能够立刻看到，我们称为可见性

·单核时代，由于所有的线程都是在同一个CPU上执行的，CPU的缓存(cache)和内存数据的一致性其实很好解决。

因为一个进程对缓存的读写操作，对于其他的线程是具有可见性的。很简单的道理，二者都作用在CPU的缓存上，当某一线程改变了内存的值后让出使用权，其他线程再访问的时候是能正确的获取到相应的值。

·多核时代就不一样了，多核导致每个CPU都有自己的cache，对于维持缓存和内存数据的一致性就变得困难了，相应的也很容易产生bug : ( 这时候不同CPU上的线程是不具备可见性的

原因：假设线程 A 和线程 B 同时开始执行，那么第一次都会将 count=0 读到各自的 CPU 缓存里，执行完 count+=1 之后，各自 CPU 缓存里的值都是 1，同时写入内存后，我们会发现内存中是 1，而不是我们期望的 2。之后由于各自的 CPU 缓存里都有了 count 的值，两个线程都是基于 CPU 缓存里的 count 值来计算，所以导致最终 count 的值都是小于 20000 的。这就是缓存的可见性问题。循环 10000 次 count+=1 操作如果改为循环 1 亿次，你会发现效果更明显，最终 count 的值接近 1 亿，而不是 2 亿。如果循环 10000 次，count 的值接近 20000，原因是两个线程不是同时启动的，有一个时差。

编译器有时候为了会优化性能，理论上会顺序执行你的高级语言程序，其实并不是。有时会调整高级语言执行的顺序却不影响程序的执行结果，但是这同时也导致了某些和设想之中不一样的bug。

例如有m、n进程调用方法getInstance来初始化实例。理论上是这么个情况：

二者都发现instance==null然后对Singleton.class上锁。但是JVM会只让一个线程成功上锁，假设是m成功上锁，这时候m就会继续发现instance==null，然后初始化instance，然后释放锁。

这时候n线程发现instance！=null了，不能上锁了，从而实现了只创建一个单例。

但是我们认为线程n在进行new操作的过程是这样的：

​ a.分配一块内存T

​ b.在T上初始化单例对象

​ c.T的地址赋值给instance

实际上却是这样的：

​ a.分配一块内存T

​ b.T的地址赋值给instance

​ c.在T上初始化单例对象

这时候问题就出来了，假设m执行完了实际上的指令b，线程切换到n，然后n检查instance是否为空，这时候instance是有T的地址的，肯定是非空，所以线程n就会直接返回instance实例。但是呢，m却还没对其初始化，这也就会出现空指针异常。

优点：

·能随从访问数组元素

·实现方法：

缺点：

考虑到保证数组元素的连续性插入和删除数据，时间复杂度是非常高的。

常见错误表述

数组和链表

wrong:

​ -链表适合插入、删除，其时间复杂度为O(1);数组适合查找，时间复杂度为O(1)

这段话其实是不正确的，错在后半部分。数组并不是查找的时间复杂度为O(1)。很简单的反例，二分查找数组中的某个元素，时间复杂度就不是O(1),而是O(logn).

right:

​ -数组支持随机访问，不需要像链表一样从头节点依次查找下去才能获取到特定索引下标的值

插入

时间复杂度分析：

​ -最好时间复杂度：O(1),插在数组的尾部

​ -最坏时间复杂度：O(n),插在数组的头部，其余n-1个全都得往后移动

​ -平均时间复杂度：O(n).

​ 设插入概率均等，则都为1/n，插入第一个需要挪n，2需要挪n-1…此时加权平均值：

​ (n+n-1+n-2+…1)/n=(1+n)/2=O(n)

删除

时间复杂度分析：

​ -最好时间复杂度：O(1),删掉数组的尾部

​ -最坏时间复杂度：O(n),删掉数组的头部，其余n-1个全都得往前移动

​ -平均时间复杂度：O(n).

​ 设删除概率均等，则都为1/n，删除第一个需要挪n，2需要挪n-1…此时加权平均值：

​ (n+n-1+n-2+…1)/n=(1+n)/2=O(n)

个人思考

对于数组删除某一个索引的值，我们可以尝试集中删除，而不是每次都要删除并迁移来保证数组的连续性。每次删除的时候都记录下被删除的数据的索引，但此时并不把其他的数据都迁移。直到数组已经没有二外的空间来存储数据时(其实是有的，只是说被删除索引的数据占的内存这时候得派上了用场）只能把理应当早就被迁移的数据迁移，让数组继续保持连续性，这就大大降低了删除的时间复杂度，而这也是JVM的垃圾回收机制的思想：标记清除算法、也是MySQL的在B+树上的删除的思想

小小拓展：JVM的GC机制算法：

​ 1.复制算法

​ 2.标记清除算法

​ 3.标记整理算法

标记清除算法具体实现： 数组中删除数据时,并不真正的删除,而是标记一下,先不进行数据的搬移工作,等数组空间不够用时,我们再执行删除操作.进行数据的搬移工作. –> 这样可以减少因为删除操作导致的数据搬移.

当然，也会导致某些离谱的现象：其实内存是有的，但是内存碎片化、不连续，最末尾的地址有了数据就被当成了没有了存储空间，从而无法防止元素情况。

容器与数组

java提供了容器类ArrayList，底层实现还是数组。

容器类的一大优势就是，把数组的基本操作(插入，删除等)封装了起来，屏蔽了许多细节，并且支持动态扩容。(小补充，ArrayList初始容量为10，每次扩容大小为扩容前的1.5倍)

但是呢，虽然能自动扩容，但最好还是每次使用前，如果能直到存储内容的大小，还是预先定义会更好。

因为扩容虽然很方便，但是其实底层涉及的操作(申请内存+搬运数据，搬运操作指的是什么？没错，就是CV大法，把数据全都复制粘贴到新申请的内存空间，是不是听起来就发现了扩容涉及的操作很耗时？)却能浪费不少时间，所以还是建议能预定义会更好。

这么一说是不是ArrayList完全取代了数组呢？

肯定不是的啦。ArrayList的局限性在于不能存储基本类型(int、long),只能先将其封装成Integer、Long。

那么问题来了，AutoBoxing(自动装箱：基本类型->类)、Unboxing(拆箱：类->基本类型)实际上也是要消耗不少时间，所以存储的是基本类型的数据的时候，最好还是考虑用数组可能会更好噢。

死锁的处理策略

发表于 2020-11-29 更新于 2020-12-04

静态策略：预防死锁

核心思想：破坏死锁的四个条件之一或几个，预防死锁的发生

破坏互斥条件

SPOOLing技术：将某互斥资源改成可同时使用的资源。

缺点：很明显，互斥性的存在是为了保护系统的安全，破坏了互斥性会危害系统的安全

破坏不剥夺条件

思想：进程所拥有的资源未使用完之前就用某种方式将其夺走，而不是等待主动释放

方法一：进程请求新的资源得不到满足时，主动释放已拥有的资源，以后再需要的时候需重新申请

​ 即使某些资源未使用完，也需主动释放。

方法二：进程需要的资源被某进程占用的时候，由操作系统协助，强心剥夺资源

缺点：

​ -实现起来较为复杂

​ -释放进程已经获得的资源会导致该进程的前一阶段工作失效。因此只适用于易保存和易恢复状态的资源

​ -反复申请和释放资源会增加系统开销，降低吞吐量

​ -方法一可能会进入这种情况：

​ 进程主动释放->进程申请->进程释放->进程申请…这就会导致饥饿的发生

破坏请求保持条件

采用静态分配方法：操作系统在进程A执行之前，把A所需要的全部资源都分配给A，且不允许A之外的进程剥夺

缺点：一直拥有很多资源，会导致资源利用率低；有时会导致其他进程饥饿

破坏循环等待条件

原理：给每个资源都编号，只能按编号从小到大的顺序来申请资源

​ 因此只会出现拥有小编号资源的进程等待拥有大编号资源的进程，

​ 而不可能出现大等小，从而破坏了循环等待条件

缺点：不方便增加设备；资源浪费；编程麻烦

死锁

概念区分

死锁：

​ 各个进程在互相等待对方的资源，导致各个进程全都堵塞，无法向前推进

饥饿：

​ 进程由于长期得不到资源，而无法向前推进。

死循环：

​ 进程执行的过程中一直执行某些语句，不跳出循环

三者异同点

共同点

​ 进程无法向前推进的现象

不同点

​ 死锁：至少有两个或以上的进程才能导致死锁。操作系统的问题

​ 饥饿：可能只有一个进程饥饿。处于饥饿的状态可能是长期得不到I/O资源导致阻塞态，也可能是得不到处理机的就绪态。操作系统的问题

​ 死循环：可能只有一个进程死循环，可上处理机运行。使用者的问题。

产生死锁的必要条件

互斥条件：

​ 被使用的资源必须互斥。因为互斥使用才会导致进程阻塞等待，能同时使用的资源不可导致死锁

不剥夺条件：

​ 进程拥有资源在未使用结束前，不允许资源被其他进程夺走，只能主动释放，不然则一直持有资源

请求和保持条件

​ 进程已经拥有了至少一个资源。

​ 发出了新的资源请求，被请求的资源满足互斥条件且已经被其他进程占有

​ 进程自己已有的资源选择保持拥有，不释放

循环等待条件

​ 存在进程资源循环等待链：链中进程拥有的资源是下一个进程的请求
